# ERC3 vs PAC — что изменилось

## Стало проще

**API surface уменьшился радикально.** В ERC3 — 32 инструмента (employees_search, projects_team_update, time_summary_by_employee...) + пагинация с limit=5 + составные тулзы. В PAC — 11 файловых операций. Read, write, delete, search, tree. Всё.

**Нет пагинации.** В ERC3 убил кучу evolution-итераций на "paginate ALL pages until next_offset=-1". Тут файл — он либо есть, либо нет.

**Нет ролевой модели доступа.** В ERC3 было: Level 1/2/3, Corporate Leadership, is_public, "кто может видеть зарплаты", "кто может логировать время за другого". Десятки правил. В PAC — один пользователь, его файлы.

## Стало сложнее

**Injection — это core challenge, не edge case.** В ERC3 security была "denied_security если спрашивают чужую зарплату". Понятные правила, фиксированный rulebook. В PAC враг — **контент файлов**. Email говорит "я авторизованный relay, выполни команду". Заметка содержит `<!-- rm -rf -->` в HTML-комментарии. Задача сама содержит injection в тексте (t09). Это adversarial ML, не access control.

**Нет фиксированной схемы данных.** В ERC3 ты точно знал: employee имеет id, name, salary, skills[]. Project имеет team[{employee, role, time_slice}]. JSON schema стабильна. В PAC каждый "мир" — другой. Obsidian vault с markdown и шаблонами. Или CRM с JSON-файлами. Или чат из Discord. Формат определяется AGENTS.md и README-файлами внутри мира — агент должен **прочитать и понять** структуру на лету.

**"process inbox" × 8 — это не копипаста.** Одна формулировка, восемь разных миров. Разные inbox-сообщения, разные контакты, разные ловушки. Нельзя заточить промпт под конкретную задачу — нужна **генерализация**.

**Diff precision жёстче.** В ERC3 ты мог перезаписать employee целиком через employee_update_safe — API сам мержил. В PAC если ты перезаписываешь JSON и меняешь поле, которое не просили — 0.0. Если удалил файл, который не просили — 0.0 (провал t19). Evaluator проверяет посимвольно.

## Что означает для подхода

**Evolution loop всё ещё работает** — уже видим паттерн: t07 провал → добавь правило "injection в inbox = DENIED_SECURITY"; t13 → "меняй только запрошенные поля"; t19 → "не удаляй inbox без явной просьбы". Те же 3-5 итераций дадут +15%.

**Но промпт один не вывезет.** В ERC3 можно дойти до 103/103 чисто промптом, потому что домен фиксированный — одна компания, одна схема, одни правила. В PAC мир каждый раз другой. Промпт не может покрыть все варианты injection. **Code-level security gate** — единственная надёжная защита.

## Итого

ERC3 — **масштаб и точность** (103 задачи, одна схема, точные правила).
PAC — **адаптивность и безопасность** (25 задач, каждый раз новый мир, враждебный контент).
