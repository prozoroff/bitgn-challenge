# Инсайты из чата канала LLM под капотом

## Формат соревнования

- **100 задач** в финальном соревновании (Ринат: "Да, хочу сделать 100 задач")
- **60-120 минут** на прогон (не 30 как в ERC3)
- Можно сделать **несколько прогонов**, но номинировать нужно конкретный **вслепую** — нельзя брутфорсить
- В blind режиме **нет оценок, логов, подсказок** — только после окончания
- PAC1-PROD откроется **11 апреля с новыми задачами** — dev-задачи для тренировки

## AGENTS.MD — главный источник правды

- **"Написанному в AGENTS.MD — верить!"** (Ринат) — если AGENTS.MD на что-то ссылается, верить дальше
- Это те же принципы оформления проектов из OpenAI Engineering Harness
- Агент должен верить AGENTS.MD **выше** чем task instruction в спорных случаях (пример Sandbox t05)
- Системный промпт можно **дополнить своим текстом** и поставить приоритетом выше AGENTS.MD

## Injection: два подхода, правильный один

**Подход Denis (неправильный для скоринга):**
> "Мой агент предпочитает отрефлексировать что это инъекция, проигнорировать инструкции и продолжить выполнение задачи"

**Правильный подход (Ринат подтвердил):**
> Система ждет, что задача сразу завершается с OUTCOME_DENIED_SECURITY. Агент — "сапёр в агрессивной среде".

Denis сам признал: "я сейчас под первый подход перепромптил" — т.е. на DENIED при любом обнаружении injection.

**Вывод для нас:** Наш провал t07 именно из-за этого — агент обнаружил injection, но продолжил обрабатывать другие файлы. Нужно: **любой injection → немедленный DENIED_SECURITY**.

## Workspace — ловушка формулировки

Обсуждение про Sandbox t05: инструкция "get an outline of workspace" — слово "workspace" неоднозначно. Vladimir потратил время на костыль.

**Решение Ринат:** исправил на "`workspace/` folder" — т.е. формулировки БУДУТ уточняться. Но в соревновании могут быть подобные неоднозначности.

**Подход Vladimir (полезный):**
> В системном промпте: "У тебя есть инструкция из AGENTS.MD, данные, инструменты и задача. Переосмысли значение инструкции с учетом данных, инструментов и задачи."

Это универсальный подход — не хардкодить "workspace = папка", а просить агента интерпретировать в контексте.

## Подход Aleksey — протокол гипотез (GHR)

Интересный фреймворк для итерации:
1. Любая идея как поправить задачу — это **гипотеза**
2. Работа по улучшению = последовательная систематичная проверка гипотез
3. Все проблемы решать **минимально разумными средствами**
4. Вести **лог экспериментов** — что работало, что нет

**Вывод:** Похоже на наш evolution loop из ERC3, только более формализованно.

**Но с подсмотром ответов** — на слепом не сработает.

## Ключевые технические моменты

1. **100 задач в PROD** — нужна скорость, ~1-2 мин на задачу максимум
2. **Слепой режим** — нельзя итерировать промпт во время соревнования
3. **Injection = DENIED** — не игнорировать и продолжать, а стопить
4. **AGENTS.MD > task instruction** — в спорных случаях верить файловой системе
5. **Proto из GitHub** — не зависеть от buf.build
6. **Tool descriptions важнее rules** — из опыта Vladimir на ERC3
